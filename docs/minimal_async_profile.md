## Assessment

**1. Scheduler + Metadata 占单步总时间百分比？**

Scheduler (0.1%) + Input Build (2.2%) = **2.3%** of single step total. GPU forward dominates at 97.6%. The CPU stages before the GPU forward (scheduler + ensure_blocks + input_build) total 2.3% of step time.

**2. 理论最多能隐藏多少时间？**

The CPU work that can theoretically run in parallel with the GPU forward (i.e., does NOT depend on the current step's GPU output): **884.6 us** per step. This includes scheduler (31.3 us avg), ensure_blocks (5.1 us avg), and input_build (848.1 us avg).

Not hideable: apply_output (9.4 us avg) and cleanup+metrics (31.1 us avg), both depend on GPU output or must wait until after the step boundary.

**3. Async Scheduling 理论最大加速比？**

If all pre-GPU CPU work is fully hidden behind the GPU forward, the theoretical per-step time becomes:

  `new_step_time = avg_step_total - theoretical_max_hideable`
  `= 38513.0 us - 884.6 us`
  `= 37628.4 us`

Speedup ratio: **1.024x** (i.e. 2.3% step time reduction).

**Important caveat**: this assumes 100% hiding efficiency, which requires that the GPU forward is always long enough to absorb the full CPU gap. If GPU forward < hideable CPU time, the excess CPU work is NOT hidden — the actual speedup is bounded by `max(GPU_time, hideable_time_remaining)`.

**4. 这个收益是否值得继续深入？**

**Yes, worth further investigation at higher batch sizes.** The CPU gap (923.0 us) is small relative to the GPU forward time (37.590 ms) at batch_size=1. However, scheduler overhead scales with the number of running sequences. A profile at batch_size=4 or 8 would reveal whether the gap grows enough to justify the pipeline complexity.

Current single-request observation: the CPU gap is 2.4% of the step. Async Scheduling is a modest optimization for this configuration, but a necessary one in production settings.

**5. 是否有迹象表明 CUDA Graph 更值得优先研究？**

**No.** The decode step has a stable execution graph (same ops each step, same number of layers, etc.), but the GPU forward time is already dominated by the model matmul operations rather than kernel launch overhead. CUDA Graph primarily eliminates Python-side launch overhead and kernel fusion opportunities. For a single request at small model scale, the launch overhead is a tiny fraction of the step.

More importantly, the current codebase's dynamic tensor shapes (block tables, slot mappings, position tensors) make CUDA Graph capture difficult without significant buffer management changes. Async Scheduling requires less invasive changes and addresses a more meaningful bottleneck for the goals of this project.
